2. Requirements

2.1. Introduction

The requirements of a stable version of Undigester are mainly functional. Some non-functional requirements, like including documentation and tests in a packaged bundle, should be accomplished as well, but the main purpose of this module is to perform the serialization of webtools-related Java objects into XML documents, and that process doesn't need to get awesome performance results, since it's part of the development stage of a project. As it's launched as an Ant task, it doesn't really affect if it takes one second more than its optimized version.

Of course, performance analisys and optimization should always be implemented. But that effort is not scheduled to be included in the first iterations of the project.

2.2. Functional requirements

The funcional requirements are described next:

  • Provide a way of transforming content, available as Java objects, to XML text.

  • Such content could be organized in a hierachical model, including Java iterators or arrays. The module has to provide support for handling them correctly. In other words, there's no restriction on which objects could be translated, but initially support of multiple children is restricted to iterators and arrays.

  • ACM-SL's preferred way of parsing XML is using Digester. That knowledge should be useful for performing the opposite operation, so Undigester should have a similar API, although direct translation between Digester rules and Undigester entities is not required.

2.3. Non-functional requirements

There're also non-functional requirements. Some of them are part of ACM-SL development standard.

  • The project should have a Developing Undigester document (usually this one), that explains each decision that was taken while developing the module.

  • The code should include Javadoc comments, and be designed using patterns as much as possible. For that, it'll use already existing ACM-SL packages.

  • The project should follow ACM-SL's folder structure.

  • Ant is the core tool for compiling, building and testing the module. Also, the project's Ant script will use the one that is shared along all ACM-SL projects.

  • JUnit usage is highly desirable. At least, it provides minimal confiability of the overall quality.

  • As all other projects at ACM-SL, Undigester should use Bugzilla to register missing features as well as eventual bugs. ACM-SL doesn't associate growing bug count with bad quality, usually the opposite, since the bugs and their state are known, and therefore can be managed.

  • Recently, we introduced Maven as a standard tool in future developments, so the project should include the files required for Maven to successfully generate all reports. The navigation.xml will be the same as other ACM-SL projects.

  • The Java source files should respect ACM-SL coding standards, and include such definition in a file called code-conventions.txt.

  • Last, but not least, as required by the GPL license, the file LICENSE.txt should be present as well.
